Project Overview

AI PDF Chatbot is a Retrieval-Augmented Generation (RAG) application that allows users to upload a PDF document and ask questions about its contents in natural language. Instead of manually searching through long documents, users can interact with the PDF through a chat interface and receive concise, context-aware answers generated by an AI model.

The project was designed around a practical use case: students and professionals often need quick answers from lengthy PDFs such as university admission guides, FAQs, policy documents, manuals, and research material. To demonstrate this, the project includes a sample university FAQ PDF built from information collected across multiple university websites.

The application combines PDF parsing, text chunking, local semantic embeddings, vector similarity search, and LLM-based answer generation. It uses local embeddings for document understanding and Groq's Llama 3.1 model for fast natural language responses.

Problem Statement

Large PDF documents are difficult to search when users do not know the exact keywords or page numbers. Traditional search only matches text literally, which often fails when a user asks a question in a different phrasing from the source document.

This project solves that problem by building a chatbot that can:

  • Extract readable text from PDF files.
  • Break long documents into meaningful overlapping chunks.
  • Convert document chunks into semantic embedding vectors.
  • Match user questions with the most relevant document sections.
  • Generate short, accurate answers using retrieved context.

The goal was to create a simple but complete AI document assistant that demonstrates the core mechanics of modern RAG systems.

Core Features

PDF Upload and Parsing

Users can upload a PDF through the Streamlit interface. The application stores the uploaded file locally and extracts text using PDFPlumberLoader from LangChain Community. This allows the system to process structured PDF content and convert it into text-based documents that can be indexed.

Text Chunking

After extraction, the PDF text is split using LangChain's RecursiveCharacterTextSplitter. The app uses chunks of 1500 characters with 200 characters of overlap. This improves retrieval quality because each chunk carries enough context while still remaining small enough for semantic search and LLM prompting.

Local Embeddings

The project uses sentence-transformers with the all-MiniLM-L6-v2 model to generate embeddings locally. This means document embedding does not require an external API call, which reduces cost and keeps the indexing step faster and more private.

The embedding model is cached with Streamlit's resource cache so it is loaded once and reused across interactions.

Semantic Search

When the user asks a question, the app converts the question into an embedding vector and compares it against all document chunk embeddings using cosine similarity from scikit-learn.

The top matching chunks are selected as retrieved context. In the main implementation, the system retrieves the top 3 most relevant chunks and passes them to the language model.

AI Answer Generation

The retrieved document context and the user's question are sent to Groq's Llama 3.1 8B model through Groq's OpenAI-compatible API endpoint.

The prompt instructs the model to:

  • Answer only from the retrieved context.
  • Keep the answer concise.
  • Use a maximum of three sentences.
  • Say that it does not know if the answer is not present in the context.

This keeps responses grounded in the uploaded PDF and reduces hallucination risk.

Upload Cache

The Streamlit app calculates an MD5 hash of the uploaded PDF bytes. If the same PDF is uploaded again during the session, the app avoids reprocessing and re-indexing it. This improves user experience by making repeated interactions faster.

Chat Interface

The main interface is built with Streamlit using st.file_uploader, st.chat_input, and chat message components. Users upload a document, wait for indexing, and then ask questions directly in the browser.

The project also includes a second prototype in the v2 folder that uses Flask with a simple HTML/CSS/JavaScript chat interface and a JSON API endpoint.

Technical Architecture

The main Streamlit version follows this flow:

  1. User uploads a PDF.
  2. The PDF is saved into the local pdfs/ directory.
  3. PDFPlumberLoader extracts document text.
  4. RecursiveCharacterTextSplitter splits the text into overlapping chunks.
  5. sentence-transformers converts chunks into normalized embedding vectors.
  6. The embeddings are stored in memory for the current session.
  7. The user submits a question.
  8. The question is embedded with the same local model.
  9. Cosine similarity ranks the document chunks by relevance.
  10. The top chunks are inserted into a prompt.
  11. Groq's Llama 3.1 model generates a concise answer.
  12. The response is displayed in the chat UI.

Main Implementation Details

The main application is contained in AI_PDF_Chatbot.py.

Important implementation choices include:

  • st.cache_resource is used to cache the embedding model and avoid repeated model loading.
  • PDF uploads are stored locally before parsing.
  • The uploaded file is hashed to prevent unnecessary re-indexing.
  • Embeddings are normalized for better similarity comparison.
  • Cosine similarity is used for lightweight in-memory retrieval.
  • Retrieved chunks are joined into a single context string before prompting the LLM.
  • The Groq API key is loaded from .env using python-dotenv.
  • Clear error handling is included for missing API keys, network errors, and API response errors.

Secondary Flask Prototype

The v2 folder contains a Flask-based version of the project. It exposes an /api/ask endpoint that accepts a user query and returns an AI-generated answer as JSON.

Unlike the main Streamlit app, the v2 pipeline is structured around a university FAQ knowledge base and includes:

  • Keyword extraction using regular expressions and NLTK stopwords.
  • University name detection from the user query.
  • SPARQL query generation.
  • Retrieval from a Fuseki endpoint.
  • LLaMA-based answer rewriting.
  • A browser-based chat interface using HTML, CSS, and JavaScript.

This version demonstrates how the project could evolve from PDF-only retrieval into a structured knowledge-base chatbot.

Challenges and Solutions

Challenge: Extracting Useful Text From PDFs

PDF files can contain inconsistent layouts, spacing, and formatting. The project uses PDFPlumber through LangChain's loader because it is reliable for extracting text from many document types.

Challenge: Preserving Context During Retrieval

If chunks are too small, answers may lack enough information. If chunks are too large, retrieval becomes less precise. The app uses 1500-character chunks with overlap to balance context and relevance.

Challenge: Avoiding Repeated Processing

Embedding generation can take time for larger PDFs. The app uses file hashing and Streamlit session state to avoid re-indexing the same PDF during a session.

Challenge: Keeping Answers Grounded

The prompt explicitly tells the model to answer only from the retrieved context and admit when it does not know. This makes the chatbot more reliable for document-based question answering.

Current Limitations

  • The vector store is currently in memory, so indexed documents are not persisted across app restarts.
  • The main version processes one uploaded PDF at a time.
  • The app depends on Groq API availability for answer generation.
  • The secondary Flask version requires external Fuseki and LLaMA endpoints to be configured before it can run fully.
  • The current UI is functional but can be improved for production use, especially on mobile devices.
